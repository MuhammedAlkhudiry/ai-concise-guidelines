# Feature Research

Research the idea before implementation. Be direct, evidence-based, and willing to challenge weak assumptions.

## Inputs

Use both:

- Codebase and product context for current behavior, constraints, and overlap
- Web research for prior art, patterns, competitors, complaints, and signals

Separate evidence, inference, and guesswork clearly.

## Output

Write one document with these headings:

1. Understanding of idea
2. Problem and context
3. Key assumptions
4. Current product reality
5. Market and patterns
6. UX direction
7. Tech and implementation options
8. Options and trade-offs
9. Recommendation and first slice
10. Suggested next steps
11. References

## What Good Research Does

- States the real problem, not just the requested feature
- Checks whether the idea duplicates or conflicts with current product behavior
- Compares multiple build options, including a smaller first slice and a credible "do not build yet" option
- Names risks, edge cases, and unknowns
- Ends with a recommendation and a cheap way to validate it

## Rules

- No code unless the user changes the task.
- Do not flatter weak ideas.
- Prefer a few strong findings over broad vague commentary.
- Only cite sources that informed the conclusion.
